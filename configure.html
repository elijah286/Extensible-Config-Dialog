Configuring this repository

These settings define what your CI does whenever a revision is submitted to this repository — whether you push directly to the default branch, open a pull request from a branch here, or receive a pull request from a fork. They apply only to the repo above, detected automatically from the dashboard you launched this from.

Pipeline settings

LabVIEW version, platforms, runner concurrency, and which activities run. Written to .github/labview-ci.yml.

Fixed to LabVIEW 2026 — the container images and tooling are built for it.

Linux container jobs run on the NI base image and are experimental — opt in only if you want them. Windows is the supported default.

Per-repo cap, drawn from your account-wide pool.
Activities

The checks and reports this pipeline can run. Choose when each one runs under When activities run below.

When activities run A “submission” is any change arriving at this repository. Standard git workflows map to three sources: a push to the default branch (a direct commit, or a merged pull request landing on the default branch), a pull request from a branch in this repository, and a pull request from a fork (an outside contributor). Tick which activities run for each source. Pull requests from forks run with a read-only token and no access to your repository secrets, so they can validate code but can't deploy or publish.

Decide what happens for each way a revision reaches this repository. For example, run Mass Compile and VIDiff on every pull request, but build the VI Snapshots gallery only when a commit lands on the default branch.

Saved to your configuration and included in the pull request. Activities already run on the sources ticked by default; custom per-source rules are recorded here and take effect as each workflow is wired to honor them — the same way Max concurrent runners is stored today. Pull requests from forks run sandboxed by GitHub — read-only and without access to your secrets.

Worker container

The default prebuilt image your CI activities run in, and the VIPC dependency files baked into it. Stored in .github/labview-ci.yml under config.container; built and pushed to GHCR by the Build LabVIEW CI Image workflow. Individual actions can override this default in Action Routing.

Latest always uses the most recently built image, so your dependencies stay current. Pin to a specific build to freeze the container for reproducibility.
VIPC dependencies baked into the container

VIPC files found in your repository are listed below, de-selected by default. Tick the ones you want baked into the worker image so their add-on dependencies are available in CI. Turn on Monitor for an included file to rebuild and re-push the container automatically whenever it changes on a submission — so the container always matches the dependencies the project needs.

Container dependencies (VIPM packages)

VIPM packages baked into the CI worker image alongside your VIPC files. Written to .github/labview/vipm/ci-tooling.packages.json; the image rebuilds when this changes.

Build / update the container now

Create a new image version or update the existing one and push it to your GHCR, on demand — without waiting for a file change. The build always publishes a moving :latest tag plus an immutable per-content tag (e.g. win-ab12cd34ef56) you can pin to. Monitored VIPC files also trigger this automatically when they change.


        

        
Tip: after it finishes, the new build appears as a pinnable tag in Default container above and in Action Routing.

      

    


    
    

      

        
Which container each action runs in

        
By default every action uses the Default container from the Container tab. Override it per action here — for example pin VIDiff to a frozen build for stable rendering while Mass Compile tracks Latest. Saved to config.container.actions in .github/labview-ci.yml; the matching Windows workflow reads it at run time.

        

        

        
Choices: Project default (inherit the Container tab), Latest build, a pinned build, the bare NI base image (no baked dependencies), or a custom tag. Linux jobs always use the NI base image.
